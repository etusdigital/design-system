Three fixes:

1. **Fix isIncluded duplicate check** (TagSelect.tsx ~line 112-123): The isIncluded function checks against the options array instead of selectedValues. For object options, the reference comparison always fails. Fix: check against selectedValues using the valueKey for comparison:
```typescript
function isIncluded(tag: any): boolean {
  const tagValue = isObject(tag) ? tag[valueKey] : tag;
  return selectedValues.some(sv => {
    const svValue = isObject(sv) ? sv[valueKey] : sv;
    return svValue === tagValue;
  });
}
```

2. **Fix tag creation flow** (TagSelect.tsx): When creating a new tag in creatable mode, the current code adds directly to selectedValues. Instead, follow the Vue pattern:
   - Create the new option object (e.g., `{ [labelKey]: inputText, [valueKey]: inputText }` for object mode, or just `inputText` for primitive mode)
   - Add to the options list (maintain a local `createdOptions` state that merges with props.options)
   - Then call toggleOption to select it
   - This ensures the new tag appears in both the dropdown options and selected tags

3. **Move icon to left side** (TagSelect.tsx line 262): Currently passes `complement={iconNode}` to SelectContainer, which renders in the ml-auto right-side div in Container.tsx. Instead:
   - Add a `leadingIcon` or `prepend` prop to the Container/SelectContainer chain, OR
   - Simpler approach: In TagSelect.tsx, instead of passing icon as `complement`, render the icon node directly before the status/tags content inside the trigger area. Look at how the trigger content is structured and prepend the icon as the first flex child.
   - If Container.tsx has a slot/prop for leading content, use it. If not, modify Container.tsx to support a `leadingComplement` prop rendered as the first child in the flex container (before label/status), with no ml-auto.
